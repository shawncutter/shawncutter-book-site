# Pads We Can Ship in Six Months

## Standard blocks, honest timelines.  
  

 ![Image](https://lex-img-p.s3.us-west-2.amazonaws.com/img/e47e387f-c672-47a5-8b47-bac775fbbc44-RackMultipart20251130-156-1pbwcz.png)

Scene: Job trailer, Mansfield. On the table: three laminated templates labeled “50 MW,” “75 MW,” and “100 MW.”

Each has:

- A known Bill of Materials.

- A known permit stack.

What’s missing is the fantasy that every site needs a completely custom reinvention of the wheel.

### Custom kills schedules

Most of the industry still treats each large energy/data campus like a boutique sculpture:

- Custom everything.

- Reinvented sequences.

- New KPIs and dashboards each time.

- “Special case” logic written into every pipe and wire.

That’s how you end up with:

- Slipped timelines.

- Weird one-off parts that take 40+ weeks to arrive.

- Teams that never get to compounding performance because they’re always in first-time mode.

We’re taking the opposite approach:

> **Pads first. Custom later.**

### Canonical pads, fully underwritten

Because we’ve already underwritten the Mansfield site—and others like it—in a **different way** (energy first, repeatable sequences, tech-assisted feasibility), we can:

- Define **canonical pad designs** :

Each pad includes:

- **Fuel** : gas intake, pressure management, basic storage assumptions.

- **CHP/heat reuse** : clear tie-points for greenhouses and process loads.

- **BESS** : sized to the block, with standard control interfaces.

- **Polite interop** : standard, open protocols for utility and downstream loads.

- **Greenhouse tie-ins** : pre-planned headers and routing, not excuses.

- **Local compute bays** : space and power carved out for local/edge workloads.

We use technology to:

- Run feasibility against:

- Move from **go/no-go** to **pad-ready** in a controlled, repeatable window.

Because we’re not chasing grants to make the numbers work, we can move on commercial timelines.

### How we measure whether the pads are real

KPI skeleton:

- **Lead-time adherence**

- **Change orders per month**

- **% repeat parts**

- **Days from pad ready → first kWh**

If those numbers improve each time we drop a pad, then the system is working.

* * *

### Rule: Standardize everything you can stomach.

Standardization feels constraining when you’re used to bespoke builds. But it’s how you earn:

- Reliable timelines.

- Bulk pricing.

- Teams that get better with each cycle instead of starting over.

> **Rule: Standardize everything you can stomach.**

Leave customization for the layers that truly require it: specific tenant configs, unique local requirements, edge cases. The bones need to be boring and reliable.

### Mini Playbook: Pads as a product

1. **Pick your block size**

2. **Publish the BoM**

3. **Assign repeat crews**

4. **Stage spares and debrief every pad**

That’s how you go from “we did a project” to “we have a product.”
