# Motel Whiteboard MVP

_Build where it runs, then keep it there._

Stale coffee. Thin towels. A whiteboard that ghosted old ideas. Prairie motel, two miles from a saltwater site. The operator said, “Show me something that makes tonight easier.”

We wrote the smallest loop that matters: poll sensor, sanity‑check value, alarm on **rate of change** , page the human who can fix it. No brand deck. No theater. We built with engine noise in our ears and field notes for specs.

Morning came. We put the laptop on a truck hood. “Hit this. When the bar moves like this, you drive. If it moves like that, you don’t.” No applause. A nod. That nod beats any congratulatory email I’ve saved.

We shipped what we proved because we **built where it runs**. After that, every feature answered one question: _How does it behave at 3 a.m. in bad weather?_ If we couldn’t answer in specifics, it didn’t ship.

**AI tie.** Today, AI keeps us honest. It turns rough voice memos into acceptance tests. It drafts docs in the operator’s language. It generates **synthetic ugly data** to make sure pretty graphs behave when the site doesn’t. Field first, forever.

_Rule: Build it where it runs—and keep it there._

**AI Playbook — Motel MVP**

**Purpose:** Convert real field pain into a testable loop in 24 hours. **Prompt:** “Write the user story in the operator’s words. (1) Define the minimal loop that makes tonight easier, (2) list boring checks before any graph, (3) require a single action a human can take at 3 a.m., (4) have AI draft docs and a 2‑minute field demo script.” **Constraints:** No feature without a 3 a.m. behavior • Logs before visuals • Next build happens on site, not in slides.
